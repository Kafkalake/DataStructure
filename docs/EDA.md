# Overall
SECS/GEM이 설비를 '제어'하고 기본적인 상태를 파악하기 위한 통신이었다면, <br>
EDA(Equipment Data Acquisition)는 오직 '초고속 대용량 데이터 수집'만을 위해 탄생한 차세대 SEMI 통신 표준입니다. <br>
반도체 업계에서는 종종 'Interface A'라고도 부릅니다. (SECS/GEM은 Interface B로 분류됩니다.)

---

### 1. EDA 표준의 등장 배경 (SECS/GEM의 한계)
기존의 SECS/GEM은 다음과 같은 한계가 있었습니다.
* 병목 현상: 설비를 '제어'하는 통신망에 '대용량 데이터 수집' 작업을 수행할 경우 네트워크와 설비 PC에 과부하 발생
* 1:1 통신: 품질 분석팀, 예지보전팀 등 복수의 클라이언트로 설비 데이터 제공 불가

이러한 문제를 해결하기 위해, 제어와 데이터 수집의 역할을 완벽히 분리하여 '오직 데이터만 초고속으로 퍼 나르기 위해' 만든 전용 고속도로가 바로 EDA입니다.

---

### 2. EDA 표준의 핵심 특징

* 데이터 전용 및 읽기 전용 (Read-Only) :<br>
  EDA로는 설비에 "가동해라", "멈춰라" 같은 제어 명령을 내릴 수 없습니다. <br>
  오직 데이터를 조회하고 구독(Subscribe)하는 것만 가능합니다. 제어는 여전히 SECS/GEM이 담당합니다.


* 멀티 클라이언트 지원 (1:N 통신) :<br>
  여러 개의 클라이언트 시스템(수율 분석 서버, FDC 서버, 예지보전 서버 등)이 동시에 하나의 설비에 붙어서 각자 원하는 데이터를 가져갈 수 있습니다.


* 최신 IT 웹 기술 기반 (HTTP/SOAP/XML) :<br>
  오래된 시리얼 통신 규격을 발전시켜 온 SECS/GEM과 달리, <br>
  EDA는 처음부터 인터넷 웹 표준 기술(HTTP, XML, SOAP 등)을 기반으로 만들어져 IT 시스템과의 연동이 수월합니다. <br>
  (최근에는 더 빠른 gRPC 규격도 도입되고 있습니다.)


* 자기 서술적 데이터 모델 (CEM: Common Equipment Model) :<br>
  EDA는 설비 스스로 설비 구조도(메타데이터)를 호스트에 제공합니다. <br>
  덕분에 데이터를 수집하는 쪽에서 설비의 물리적 구조를 쉽게 이해하고 원하는 센서만 콕 집어서(DCP: Data Collection Plan) 구독할 수 있습니다.

---

### 3. SECS/GEM vs EDA 비교 요약

| 구분 | SECS/GEM (Interface B) | EDA (Interface A) |
| :--- | :--- | :--- |
| **주요 목적** | 설비 **제어** 및 기본 상태 모니터링 | 대용량 센서 데이터 **초고속 수집** |
| **통신 방식** | 1:1 연결 (Host - Equipment) | 1:N 연결 (Multiple Clients - Equipment) |
| **데이터 방향** | 양방향 (명령어 전송 가능) | 단방향 (읽기 전용, 데이터 수집만) |
| **적용 기술** | HSMS (TCP/IP 기반 자체 규격) | HTTP, SOAP, XML, Web Service |
| **비유** | 공장의 '리모컨' 및 '무전기' | 공장의 '고속 CCTV' 및 '블랙박스' |

### 결론
현대의 첨단 반도체/디스플레이 공장(Fab)에서는 이 두 가지 표준을*함께 사용합니다. <br>
공장의 메인 시스템(MES)은 SECS/GEM을 통해 설비에 작업 지시를 내리고 공정(Step)을 제어하며, <br>
품질 분석 시스템(FDC, APC)은 EDA 고속도로를 통해 해당 공정 중 발생하는 수천 개의 센서 데이터를 <br> 
실시간으로 빨아들여 불량을 감지하고 수율을 높이는 아키텍처를 구성합니다.
